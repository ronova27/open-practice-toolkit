# Branch Plan

Create each workstream branch from `main`. Each branch starts with a small branch note so it can be opened independently in Codex or reviewed as a planning lane.

## Main

`main` contains the founding documentation, governance files, templates, workstream map, and application notes.

## Workstream Branches

1. `workstream/product-foundation`
   - Expand user stories, glossary, product principles, and contribution framing.

2. `workstream/metronome-engine`
   - Define timing architecture, scheduler assumptions, data model, and accuracy risks.

3. `workstream/notation-prehear`
   - Compare notation rendering/playback approaches and export options.

4. `workstream/practice-feedback`
   - Plan recording, onset detection, latency compensation, and visual timing feedback.

5. `workstream/audio-to-notes`
   - Plan monophonic transcription, manual correction, and MIDI export.

6. `workstream/video-to-rhythm`
   - Plan experimental motion timing analysis and local-first privacy constraints.

7. `workstream/web-pwa-shell`
   - Plan the initial frontend shell, offline behavior, storage, and accessibility baseline.

8. `workstream/docs-and-governance`
   - Refine project governance, issue templates, security policy, and maintainer workflows.

## Merge Expectations

- Workstreams should be merged only after focused review.
- Branches may intentionally contain research notes before implementation.
- Any future application scaffold should be introduced in a small, explicit branch.
